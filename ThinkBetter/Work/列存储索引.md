**列存储索引必须与基表分区对齐。请使用与基表相同的分区函数和相同（或等效）的分区方案创建列存储索引。如果基表未分区，则创建非分区列存储索引。**

**重建列存储索引时不允许使用 ONLINE 选项**

分区功能允许我们对同一张表使用不同的列存储压缩算法
```sql
alter Index PK_FactOnlineSales on dbo.FactOnlineSales
	REBUILD PARTITION = 2   --
		WITH (DATA_COMPRESSION = COLUMNSTORE_ARCHIVE);
```


聚集列存储索引更新操作并没有采用批处理模式


#SQLServer/2014  中新增了一个属性——**有效并行度**，它被描述为“_列存储索引构建期间的最大并行度_”；减少内存量，SQL Server 将逐步降低 MAXDOP 值，直到降至 1，从而有效地采用非并行处理。

实际的构建过程大致分为两个阶段：第一阶段只使用一个可用核心——顾名思义，这个阶段用于创建字典；第二阶段则使用所有可用/指定的 CPU 核心来创建实际的聚集列存储索引。



**批处理模式查询**是在 SQL Server 2012 中引入的，专门用于列存储索引。以 1000 个数据单元为单位进行数据连接处理，而不是像传统行模式那样逐个处理，从而保证了处理海量数据时更高的 CPU 效率。使用CTE方式的写法，可以到批处理模式。

### TempDB 溢出 – 当内存不足时
最常见的原因是统计信息估计错误——导致对必要内存的请求实际上是在请求错误的值。在这种情况下，为了保证查询能够继续执行，数据会被传输（溢出）到磁盘，以避免查询中断。当前数据会被移动到 TempDB 数据库，以便为新数据腾出空间。每当需要再次读取溢出的数据时，都会从 TempDB 磁盘读取。整个过程极其缓慢，因为我们没有直接操作内存，这也是为什么有些查询执行速度极慢的原因——它们不断地读写磁盘，而没有明确说明具体的读写过程。
