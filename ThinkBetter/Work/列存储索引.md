## 元祖移动器（**Tuple Mover**）
当我们向表中插入新行（即_增量存储_）时，在达到一定数量（普通插入 100 万行，批量插入 10 万行）后，增量存储将会被填满，并关闭以停止任何额外的数据插入。SQL  
Server 有一个自动后台进程，它会检查已关闭的_增量存储_（状态=2），并在发现后通过编码和压缩将其转换为_段_。这个后台进程称为元组移动器，默认情况下每 5 分钟执行一次。元组移动器的设计目的是不阻塞任何数据读取操作，但并发的删除和更新操作必须等待压缩过程完成。我认为这应该不会造成太大的阻塞，因为我们讨论的并非 OLTP 系统。  
**注意**：实际上，如果在重建表时，已压缩的_段_（_行组_）尚未达到最大行数（约 100 万行），则可以向其中添加额外的行。更多内容将在后续文章中介绍。  
**注意**：始终可以通过执行`alter index ... reorganize`  
以下语句来调用元组移动器：调用以下语句会重建整个表，因此不会调用元组移动器：  
`alter table ... rebuild`或  
`alter index ... rebuild`。


## 批量插入

批量插入操作的处理方式取决于插入的行数。100,000 行是分隔符的上限，因此，假设我们要插入 85,000 行，它们将被添加到现有或新的_增量存储中_；但如果我们要插入 120,000 行，它们将自动转换为_段_（行组）。**理想的插入行数应超过 100 万行，因为这样会自动转换为不可更新的完整_段_**。  
该操作本身非常高效，因为它会将所有插入的行存储在内存中，并在内存中应用编码和压缩，只有在进程终止后才会将数据存储到磁盘上。此操作成功执行的自然要求是拥有足够的内存 :)  
注意：`insert into ... select from ...;` 该语句针对 SQL Server 2014 进行了优化，可使用并行批处理模式执行批量插入。

## 隔离级别

目前除快照 (Snapshot) 隔离级别外，所有隔离级别均已实现并支持与聚集列存储索引 (CSCOI) 一起使用。这意味着支持以下隔离级别：READ UNCOMMITTED、READ COMMITTED、REPEATABLE READ 和 SERIALIZABLE。  
此外，还支持 READ_COMMITED_SNAPSHOT 隔离级别。

值得思考的是，微软为何几乎实现了所有隔离级别，却放弃了快照 (SNAPSHOT) 隔离级别，而 READ_COMMITED_SNAPSHOT 隔离级别显然并非数据仓库中最常用的隔离级别。这是否意味着未来将不再实现/支持快照 (SNAPSHOT) 隔离级别？


## 锁定

聚集列存储索引的锁定是在_页面_级别进行的（这是微软公布的）
但是，测试表明，即使是简单的操作也会在相应的_行组_上获取 IX（意向排他）或 X（排他）锁（插入操作也适用于_增量存储_），从而导致无法从中读取数据。
