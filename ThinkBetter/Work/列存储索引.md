## 元祖移动器（**Tuple Mover**）
当我们向表中插入新行（即**增量存储**）时，在达到一定数量（普通插入 100 万行，批量插入 10 万行）后，增量存储将会被填满，并关闭以停止任何额外的数据插入。SQL  
Server 有一个自动后台进程，它会检查已关闭的**增量存储**（状态=2），并在发现后通过编码和压缩将其转换为**段**。这个后台进程称为元组移动器，默认情况下每 5 分钟执行一次。元组移动器的设计目的是不阻塞任何数据读取操作，但并发的删除和更新操作必须等待压缩过程完成。我认为这应该不会造成太大的阻塞，因为我们讨论的并非 OLTP 系统。  
**注意**：实际上，如果在重建表时，已压缩的**段**（**行组**）尚未达到最大行数（约 100 万行），则可以向其中添加额外的行。更多内容将在后续文章中介绍。  
**注意**：始终可以通过执行`alter index ... reorganize`  
以下语句来调用元组移动器：调用以下语句会重建整个表，因此不会调用元组移动器：  
`alter table ... rebuild`或  
`alter index ... rebuild`。


## 批量插入

批量插入操作的处理方式取决于插入的行数。100,000 行是分隔符的上限，因此，假设我们要插入 85,000 行，它们将被添加到现有或新的**增量存储中**；但如果我们要插入 120,000 行，它们将自动转换为**段**（行组）。**理想的插入行数应超过 100 万行，因为这样会自动转换为不可更新的完整**段****。  
该操作本身非常高效，因为它会将所有插入的行存储在内存中，并在内存中应用编码和压缩，只有在进程终止后才会将数据存储到磁盘上。此操作成功执行的自然要求是拥有足够的内存 :)  
注意：`insert into ... select from ...;` 该语句针对 SQL Server 2014 进行了优化，可使用并行批处理模式执行批量插入。

## 隔离级别

目前除快照 (Snapshot) 隔离级别外，所有隔离级别均已实现并支持与聚集列存储索引 (CSCOI) 一起使用。这意味着支持以下隔离级别：READ UNCOMMITTED、READ COMMITTED、REPEATABLE READ 和 SERIALIZABLE。  
此外，还支持 READ**COMMITED**SNAPSHOT 隔离级别。

值得思考的是，微软为何几乎实现了所有隔离级别，却放弃了快照 (SNAPSHOT) 隔离级别，而 READ**COMMITED**SNAPSHOT 隔离级别显然并非数据仓库中最常用的隔离级别。这是否意味着未来将不再实现/支持快照 (SNAPSHOT) 隔离级别？


## 锁定

聚集列存储索引的锁定是在**页面**级别进行的（这是微软公布的）
但是，测试表明，即使是简单的操作也会在相应的**行组**上获取 IX（意向排他）或 X（排他）锁（插入操作也适用于**增量存储**），从而导致无法从中读取数据。


## 内存消耗

 SQL Server 2014 中改进了内存消耗控制。因此，除了传统的并行度 (DOP) 控制之外，我们可以预期，实际的并行度 (DOP) 会根据 SQL Server 当前可用资源的变化而变化。这意味着为了保持系统稳定，某些线程甚至可能会被暂停。



## 归档压缩

在数据仓库环境中，经常会存在一些很少访问的数据子集（历史数据），但它们却占用着相当大的空间。SQL  
Server 2014 为列存储索引引入了一种新的压缩模式，称为**归档压缩**，以解决这一问题。此模式在已有的压缩层之上增加了一个额外的压缩层。它通过在数据存储到磁盘之前压缩数据块来透明地工作。  
压缩算法采用 XPress8（微软在 20 世纪 70 年代开发的[LZ77 压缩](http://en.wikipedia.org/wiki/LZ77**and**LZ78)算法的内部变体）。它旨在同时处理多个线程，并支持最大 64KB 的数据流。  
与传统的列存储压缩相比，此模式的性能提升约为 30%。  
**注意：** 压缩可以应用于分区级别，因此可以确保仅对访问频率最低的数据应用此方法。可以在重建时轻松更改压缩类型。


## 字典

列存储索引中的某些列可能使用字典编码以提高压缩率。字典有两种类型：本地字典和全局字典。本地字典仅在同一**段**  
中使用， 而全局字典则应用于所有**段**。**全局字典中存储的相关数据越多，列存储索引的压缩效果就越好。** 

注意：如果全局字典完全相关，则可能无需任何本地字典。注意：**段** 内的数据永远不会排序。 编码和压缩完成后，字典将以 BLOB 的形式存储在磁盘上，并使用传统的 8KB 页机制。用于映射字典值的 ID 为 32 位。在 SQL Server 2014 中，构建全局字典的整个过程都进行了改进。首先，对所有**段**中需要字典的列 进行数据采样，并基于此分析创建全局字典。


## 统计信息采样

SQL Server 查询优化器使用统计信息来创建查询执行计划，关于统计信息的基础知识在互联网上随处可见。对于非聚集列存储索引，没有实现统计信息采样——这在某种程度上是合理的，因为不允许更新。  

聚集列存储索引则截然不同——它们有两种不同的采样算法： 

第一种算法侧重于准确性，但会消耗更多的 I/O 和 CPU 资源。该算法会扫描每个_段_和_增量存储_，然后随机选择一些行子集。与页采样相比，该算法应该能够生成更精确的直方图。  

第二种算法以性能为目标，同时降低了一些准确性。该算法使用聚集采样，通过在每个聚集组中选择一些随机行（行数由采样百分比控制）。该算法可能只扫描部分段_和_增量_存储_，而完全忽略其他部分。该算法用于辅助创建字典，但不会用于构建直方图。
